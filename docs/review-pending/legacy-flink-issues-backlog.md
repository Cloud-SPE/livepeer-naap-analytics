# Issues Backlog

Canonical backlog for unresolved feature ideas, design hardening, and engineering improvements.

## Scope and Usage

- Source of truth for open issues discovered in architecture, data contract, quality, and consolidation docs.
- Keep items implementation-oriented and traceable to source docs.
- When closing an item, update this file and the linked canonical doc in the same PR.

## Prioritization Scale

- `P0`: blocks correctness or release readiness.
- `P1`: high-value reliability or contract hardening.
- `P2`: useful enhancement; not release-blocking.
- `P3`: maintenance/documentation/process improvement.

## Parallelism Hardening Baseline (2026-03-14)

This codebase now includes the following baseline changes tied to the parallelism hardening program:

- `BACKLOG-024`: lifecycle operators wait up to 15 seconds for expected capability context before first emission.
- `BACKLOG-030`: dedup identity and fallback session identity are source-aware.
- `BACKLOG-031`: capability broadcast state is scoped by `source_namespace + wallet_address`.
- Capability dimensions now preserve both Flink lineage (`source_event_uid`) and upstream provenance (`source_event_id`) so parallelism/correction diagnostics do not overload a single event-identity field.
- `BACKLOG-032`: replay defaults to `events.replay.streaming_events.v1` instead of the live input topic.
- `BACKLOG-004`: lifecycle/status/trace delayed-attribution counters are emitted from Flink operators.
- `BACKLOG-020`: capability broadcast state moved away from a `final` map field that triggered GenericType fallback risk.
- `BACKLOG-029`: canonical latest-row views and correction-safe serving rollups now back status/trace-derived performance paths.

Remaining open work in the backlog should be read as follow-on hardening beyond this baseline, not as absence of the above protections.
